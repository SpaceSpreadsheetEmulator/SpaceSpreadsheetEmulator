# Build 3396210 login and initial-session behavior

This specification records client-visible behavior implemented from independently
authored fixtures and private black-box observations. It contains no captured
credentials, session material, or third-party implementation code.

## Version and crypto negotiation

The server first sends a marshal tuple containing birthday `170472`, Macho version
`496`, user count, version `24.01`, build `3396210`, project `V24.01@ccp`, and null.
The client returns the same compatibility identity. A mismatch is rejected before an
account or session is created.

The following command is either queue check `(ignored, "QC")` or version-key
`(ignored, "VK", bytes)`. Queue check returns the current queue position and restarts
version exchange. Version-key advances to a two-item crypto request.

Two crypto-request forms are supported:

- `("placebo", {})` continues with ordinary framed marshal payloads;
- a parameter dictionary containing `crypting_sessionkey` and
  `crypting_sessioniv` selects AES-256-CBC with PKCS#7 padding. Raw 32/16-byte
  material and observed bounded 256/512-byte placebo containers are accepted; the
  AES key and IV are their first 32 and 16 bytes.

Inbound and outbound CBC chaining are independent. After each frame, that direction's
IV becomes the final ciphertext block. Invalid lengths or padding close only that
connection. The server acknowledges negotiation with `"OK CC"` before either mode
begins.

## Authentication and session

The login tuple contains a dictionary with `user_name`, `user_password_hash`, language,
and optional country metadata. The protocol adapter forwards the opaque proof to the
Worker and never logs it. Development enrollment may bind the first proof for a new
name; it is disabled by default.

On success the server sends password version `2`, then a four-item challenge tuple. Its
function slot contains a fabricated nine-byte Python-marshal string expression for
`None` (`74 04 00 00 00 4e 6f 6e 65`) and does not execute server-supplied behavior.
The final acknowledgement contains the account, client, session, language, country,
role, and peer address fields required to initialize the client session.

## Initial MachoNet traffic

Call requests use numeric type `6`; correlated responses use type `7`, swap routing
direction, retain the call identifier, and carry a marshal substream result. The first
implemented calls include service discovery, server time, and the minimal character
selection query. Ping request type `20` receives type `21` with bounded timing rows.

`charUnboundMgr.GetCharacterSelectionData` is always authored from the authenticated
session and the Worker response. It is never eligible for captured-response replay.
The result is a four-item list of `CRowset` objects:

1. one account row with user name, three character slots, null subscription end, and
   the authentication timestamp as a FILETIME;
2. an empty training rowset;
3. zero to three packed character rows with the build-3396210 column order and types;
4. an empty character-war rowset.

The current character projection supplies the Worker-owned identity, location, ship
type, corporation, balance, skill points, and last-login time. Fields outside the
starter slice use explicit neutral or null values rather than captured account data.
The identity-dependent `config.GetMultiOwnersEx` and
`config.GetMultiCorpTickerNamesEx` follow-up calls are authored from that same
validated selection. The temporary corporation ticker is derived from the
server-authored corporation name; logo fields remain null.
Gateway rejects an account mismatch, too many characters, duplicate/non-positive
identities, invalid numeric ranges, or a non-integral balance by returning the typed
rowsets with no selectable characters. Logs record only the count or validation
failure, not character names or identifiers.

Selecting a returned character emits a type-16 client-session notification before
the correlated selection response. The payload contains the deterministic starter
character, ship, station, corporation, headquarters, and map hierarchy. The minimal
movement slice recognizes the observed ship and solar-system bind calls plus the
bound `CmdDock` call. Gateway resolves the solar-system owner through Coordinator,
sends epoch-fenced mutations directly to that Worker, and emits station session
changes after successful undock and dock transitions. Unknown RPC methods still
return null; they are not interpreted as game commands.

The fresh graphical checkpoint accepts both the authored selection and the
post-selection station session packet. It continues through dozens of startup calls
and successful pings. Two later transport failures were isolated and corrected:
the saved-value tail table inside `eventLog.LogClientStats`, then the valid type-12
`ClientHasReleasedTheseObjects` notification. Neither was a rejection of the
session notification. See `station-entry-3396210.md` for the exact ordering,
failure boundaries, marshal rules, and graphical acceptance gate.

The loopback suite independently proves select/undock/movement/dock through real
Gateway, Coordinator, and Worker processes. It also restarts the topology while the
character is in space, reloads the PostgreSQL-backed snapshot, reconnects with a
stationless character selection, rebinds the solar system, and docks. This remains
server-side regression evidence until the graphical station checkpoint documented
separately is complete.
